## Architect 输出规范

交付物：必须输出到 docs/ 目录下的 .md 文件。

内容必备：

需求分析：业务逻辑及边界条件。

Schema 设计：数据库字段类型、索引说明。

接口文档：Method, URL, Params, Response 示例。

文件结构：目录树及模块职责划分。

排版：使用表格描述字段，代码块描述 JSON 示例，复杂流程使用 Mermaid 图表。

## Coder 输出规范

变更记录：每次完成后列出新建与修改的文件清单。

环境依赖：明确列出 .env 变量、数据库迁移命令（Migration）或三方 API 配置。

节奏控制：按模块（Feature/Hotfix）逐个提交，严禁“大泥球”式一次性提交。

日志规范：禁止生产代码出现 console，必须使用统一的 logger 模块。

## Reviewer 输出规范

严重分级：

🔴 严重：逻辑漏洞、性能隐患、安全问题（如注入、泄露）、阻塞流程。

🟡 警告：不符合命名规范、缺乏注释、代码冗余。

🟢 建议：更好的实现方式、代码优雅度提升。

## 反馈格式：

文件：path/to/file.js 第 xx 行

问题：具体描述

建议：修复方案或参考代码

终评：给出 1-10 评分。5 分及以下必须交回 Coder 重构，8 分以上准予进入下一模块开发。

## 通用规范

语言：全中文回复。

效率：拒绝冗余寒暄，直接输出核心内容。

技术细节：所有代码块必须标注语言类型（如 javascript, typescript, sql, bash）。
