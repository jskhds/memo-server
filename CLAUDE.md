# Flashcard Mini Program

微信小程序闪卡应用，SM-2 复习算法。前端 React + TS，后端 Node.js。当前项目是后端。

## 信任优先级(最重要)

- 代码 > 文档。文档和代码冲突时以代码为准。
- 具体实现看源码,文档只用于理解设计意图和业务背景。
- 发现文档明显过期,告诉我,我会更新。

## 技术栈

- 框架: Node
- 平台: 微信小程序 Taro

## 项目文档

- 目录： `./docs/*.md`
- 产品说明: `./docs/overview.md`
- 未决问题: `./docs/todo.md`

## 开发计划

- 当前开发计划: `../docs/plan.md`(包含前后端,你只需关注标记为后端的 step)
- 涉及前后端交互的 step,先向我确认接口约定,不要自行推测
- 历史开发计划: `../docs/plan/*.md`(供参考,不必主动读)

## 产品需求

- 各功能的 PRD 在 `../docs/prd/{功能域}/`。
- 改动涉及业务规则时,先读对应 PRD;
- 新增或修改业务规则时,提醒我更新 PRD。

## 代码约定

- 代码输出规范参考 `.claude/rules/code-rules.md` 和常见的 js 和 ts 以及 node 代码规范

## 常用命令

- /architect — 需求分析、数据库设计、接口规划，输出文档，不写代码
- /coder — 根据架构师的设计专注写代码，不做需求讨论
- /reviewer — 审查代码质量、安全性、接口一致性，给出修改意见

## 开发规范

- 所有接口统一返回格式：{ code, message, data }
- 错误处理必须完整
- 代码必须有注释
- 文件结构按功能模块划分
